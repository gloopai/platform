# 分模块开发计划（对齐「支付产品 / 上游通道」与各端）

本文在 [`开发计划.md`](./开发计划.md)、[`产品.md`](./产品.md) 与 [**通道与支付产品**](./通道与支付产品.md) 基础上，按 **后端 → 网关 → 管理台 → 商户台 → 收银台** 的顺序，列出可**小步快跑**的模块与功能点。实施时建议：**完成一项、联调一项、再开下一项**。

**核心前提**：对外统一 **支付产品**（微信、支付宝等）；对内多条 **上游通道** 由路由分配——详见 [`通道与支付产品.md`](./通道与支付产品.md)。

---

## 阶段 A：数据模型与路由内核（后端优先）

**目标**：数据库与领域模型能表达「产品 / 通道 / 绑定 / 路由结果」，订单可记录最终 `channel_id`。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| A1 | 库表 DDL | `pay_product`（或等价）支付产品；`channel` 上游通道；**产品与通道关联表**（多对多、权重、启用状态）；`orders` 增加/确认 **`channel_id`**、**`pay_product_id`**（或产品编码） | 可插入：同一微信产品下挂 2+ 条通道 |
| A2 | 路由服务（逻辑层） | 输入：商户、金额、支付产品编码；输出：**一条** `channel_id` + 可选路由原因码；预留成功率/权重/商户规则扩展点 | 单元测试：同产品多通道下可重复、可观测 |
| A3 | 熔断 / 健康占位 | 通道级「暂时不可用」标记；路由时跳过（可先写内存/DB 字段，再接 Redis） | 手动置灰某通道后，路由落到其他通道 |
| A4 | 与现有下单链路对接 | `CreateOrder` / 支付发起路径：先解析**支付产品**，再调路由，再落单 | 订单表能查到选用通道 |

**✓ 2026-03-22**：A1/A2/A4 已落地（见 `deploy/sql`、`trade` 路由、`gateway` 下单与终端 `pay_products`）；A3 仍可用 `channels.fuse_enabled` 跳过通道；管理台产品/绑定 CRUD 见阶段 C。

---

## 阶段 B：网关与开放 API（商户侧契约）

**目标**：商户只传**支付产品**，不传内部通道 ID；响应不泄露多条上游。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| B1 | 下单 API | 请求参数使用 **产品编码**（与收银台一致）；服务端内部调 A2 选路 | 与文档字段一致 |

**✓ 2026-03-22**：`pay_type` 必填；`channel_id` 不再参与商户侧选路。
| B2 | 查单 / 回调 | 对外状态仍以平台订单为准；内部可记录通道用于对账 | 回调 payload 不包含「供应商列表」 |
| B3 | 错误码 | 无可用通道、产品关闭等**产品层**错误与**通道临时失败**区分（可先简化） | 商户可理解 |

---

## 阶段 C：管理后台（Admin）—— 配置与可观测

**目标**：运营能管「产品」「通道」「绑定与路由」，并能按产品/通道看问题。

**技术栈**：现有 `frontend/apps/admin`（Vue3 + Tailwind）。

**✓ 2026-03-22**：C1（支付产品 CRUD）与 C3（产品与通道绑定、权重、启用）已落地：网关 `/v1/admin/pay_products*`、`/v1/admin/pay_product_bindings*`；页面 `merchant-products` → `PayProductsPage.vue`。C2 通道管理此前已存在。**C4** 见下表与「C4 补充」。**✓ 2026-03-23**：**C5** 简版监控（`/stats` + `GET /v1/admin/stats/overview`）。**C6** 仍待迭代。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| C1 | 支付产品管理 | 列表 / 新建编辑 / 启用禁用 / 排序 / 编码唯一 | 与库表一致 |
| C2 | 上游通道管理 | 列表 / 编辑网关、密钥、成本标记、健康状态（手动） | 敏感字段脱敏展示 |
| C3 | 产品 ↔ 通道绑定 | 为某产品勾选多条通道；**权重**或优先级；启用 | 保存后路由 A2 可读 |
| C4 | 路由策略（迭代） | 第一期：**权重**（`pay_product_channels`）+ **商户支付产品白名单**（`merchant_pay_products`，严格/开放模式，见 `docs/通道与支付产品.md` §6）；后续：成功率、熔断阈值 | 与 A2 联动；**✓ 2026-03-23**：白名单与终端/开放 API 行为已对齐 |
| C5 | 监控看板（简版） | 按 **支付产品** 交易量、成功率；按 **通道** drill-down | 可定位「微信产品下通道乙失败率高」；**✓ 2026-03-23**：今日聚合 + 产品/通道双表（`orders` 自然日），接口与 `/stats` 见 `开发日志` |
| C6 | 原有模块衔接 | 商户、渠道、费率等与「产品/通道」字段对齐，避免两套概念冲突 | 文档与界面用语统一 |

### C4 补充：管理台「路由策略」页与后续规则引擎（**说明页已做，规则引擎后面再做**）

| 子项 | 说明 | 状态 |
|------|------|------|
| **C4-ui** | 管理台 `/routing`：说明当前加权路由、白名单、熔断如何落地；卡片跳转「支付产品与通道」「商户」「通道」；`GET /v1/admin/routing/summary` 展示库表汇总 | **✓ 2026-03-23**（见 `docs/开发日志.md`） |
| **C4-rules** | **路由规则引擎（待做）**：规则表或配置中心；维度如金额段、时段、商户等级、地区等，覆盖/追加默认权重或指定主备通道；`trade` 选路前执行规则 | 未开始；与 A2 扩展点联动 |
| **C4-metrics** | **成功率 / 延迟驱动调度（待做）**：聚合近期通道指标，自动降权、恢复；与 Redis/时序库选型相关 | 未开始 |
| **C4-ops** | **与监控 / 告警联动（待做）**：人工切换、切换流水审计；依赖 C5 与通道健康数据 | 未开始 |

> 实施顺序建议：先 **C5 简版监控** 或 **A3 熔断策略增强**，再启动 **C4-rules**，否则缺少可观测性难以验证规则效果。

## 阶段 D：商户台（Merchant Console）

**目标**：商户理解与联调**支付产品**；不暴露内部通道列表（默认）。

**技术栈**：现有 `frontend/apps/merchant`。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| D1 | 开发者文档 / 常量 | 下单参数说明：**支付产品编码**与平台约定一致；说明「同一微信可能由平台路由不同上游」**一句即可** | 与网关 B1 一致；**✓ 2026-03-23**：`merchant/config/payProducts.ts` + `DevelopersPage` 文案 |
| D2 | 开发配置 / 联调 | 创建订单时选择 **产品**（下拉来自配置或写死枚举与后端同步）；展示返回的 `checkout_url` | 不展示 `channel_id` 选择器；**✓ 2026-03-23**：联调页 `pay_type` 下拉 + 自定义 |
| D3 | 交易 / 财务展示 | 列表以订单为主；可选列「支付方式」显示**产品名**；高级需求再考虑「通道级」脱敏信息 | **✓ 2026-03-23**：订单列表/详情 `pay_product_name`（网关聚合 `pay_products` 展示名）+ 编码副行 |
| D4 | 控制台指标（可选） | 成功率、金额按 **产品** 维度聚合（数据来自后端接口） | 与「产品」口径一致 |

---

## 阶段 E：收银台（Checkout）

**目标**：用户只选 **支付产品**；调起/扫码由服务端返回；界面与商户台、管理台**产品命名一致**。

**技术栈**：现有 `frontend/apps/checkout`。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| E1 | 支付方式列表 | 数据来源：**后端按订单返回可用支付产品列表**（或配置 + 订单约束），**不要**写死多条重复「微信」 | 同一产品只一项 |
| E2 | 发起支付 | 提交 **产品编码** 调网关；收二维码 / 跳转 URL / SDK 参数 | 与 B1、路由一致 |
| E3 | 环境检测（迭代） | 微信内优先微信产品等（与产品.md 一致） | 不可用产品隐藏或置灰 |
| E4 | 品牌化（后期） | 商户 Logo、主题色；不改变「产品 / 通道」逻辑 | 配置来自商户或订单 |

**✓ 2026-03-22**：E1 — `GET /v1/terminal/order` 返回 `pay_products`；收银台优先渲染列表。

**✓ 2026-03-22**：E2 — `POST /v1/terminal/pay`（trade `PrepareTerminalPay`）+ 收银台调用；返回 `pay_url` / `qr_payload` / `pay_mode`。

## 阶段 F：对账、结算与消息（与通道强相关）

**目标**：账务与订单能关联到 **通道**，对账按上游维度拉取；对外报表仍可按 **产品**。

| 序号 | 模块 | 开发内容 | 验收要点 |
|------|------|----------|----------|
| F1 | 对账文件 | 按 `channel_id` 拉取或映射上游账单 | 与订单通道一致 |
| F2 | 差错处理 | 补单、挂账与通道关联 | 可追溯 |
| F3 | NSQ / 回调 | 现有链路保持；订单状态变更带通道信息入库 | 与 A4 一致 |

---

## 建议执行顺序（小步快跑）

1. **A1 → A2 → A4**（模型 + 路由 + 下单落通道）  
2. **B1**（API 只认产品编码）  
3. **C1 → C2 → C3**（管理端能配产品、通道、绑定）  
4. **E1 → E2**（收银台跟产品走）  
5. **D1 → D2**（商户台联调与文档）  
6. **C5、A3**（可观测与熔断增强），再视需求启动 **C4-rules / C4-metrics**（路由规则引擎与动态调度，见上表）  
7. **F1～F3**（财务与对账深化）

每一步完成后：**写一条 `开发日志.md`、更新相关 `*-platform.md`**，便于下一轮 AI 接着做。
