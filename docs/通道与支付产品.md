# 支付产品与上游通道（核心产品逻辑）

本文记录聚合支付中台的一条**关键产品/架构约定**：**面向商户与终端用户的是「支付产品」**；**平台内部落地执行的是「上游通道」**。二者是 **一对多** 关系，调度策略在平台侧完成，对商户与收银台用户**屏蔽上游差异**。

---

## 1. 两个层次分别是什么

| 层次 | 名称 | 谁看见 | 含义 |
|------|------|--------|------|
| 对外 | **支付产品**（Payment Product） | 商户 API、收银台、商户台文案与能力说明 | 用户心智中的支付方式：**微信支付、支付宝、云闪付、网银** 等。同一类产品在界面上只出现一次。 |
| 对内 | **上游通道**（Upstream Channel） | 平台运营、风控、技术；**不**要求商户理解 | 与具体供应商/机构对接的实例：不同商户号、密钥、网关 URL、协议版本。**多个通道可以同属一个支付产品。** |

**举例（重要）**

- 对外只展示：**微信支付**。
- 后台可能配置：**微信-通道甲**（供应商 A）、**微信-通道乙**（供应商 B）、**微信-通道丙**（备用）——三者都是「微信支付」这一支付产品下的不同上游实现。
- 一笔订单在实际支付时，只会走**其中一条**上游通道；具体走哪条由平台**路由策略**决定。

---

## 2. 为什么要这样设计

1. **稳定性**：单通道故障时，可在**同一支付产品**内切换其他上游，而无需商户改接口或用户重新理解「另一种微信」。
2. **成本与质量**：可按费率、成功率、延迟对多条上游排序或加权，实现**智能调度**。
3. **商户体验**：商户侧只对接「微信 / 支付宝 / …」的统一产品编码与费率展示；**不要求**商户为每个上游单独签约（除非商务上单独约定，那是合同层，不是 API 层暴露多名字）。
4. **运营可扩展**：新增一家上游「还是微信支付」时，在管理后台增加通道并挂到支付产品即可，**收银台与标准 API 的产品列表可保持不变**。

---

## 3. 路由与分配（平台内部）

下列因素可用于**自动或半自动**选择具体上游通道（实现阶段可逐项上线）：

- **成功率 / 健康度**：近期失败率、超时率；可配合熔断、降级。
- **商户维度**：指定商户走指定通道、白名单、差异化费率。
- **成本**：在同等稳定性下优先更低费率通道。
- **权重与轮询**：简单负载或按比例分流。
- **地域 / 限额 / 产品能力**：如通道仅支持某额度范围或某业务类型。

路由结果应落在订单或支付单上（例如记录 **`channel_id`**），便于对账、追责与统计。

---

## 4. 各端如何体现（与实现约束）

### 4.1 开放 API / 商户侧

- 商户请求中的 **支付方式 / 产品** 应指向 **支付产品编码**（如 `pay_type`、产品码），**不**暴露内部 `channel_id` 列表。
- 查单、回调、对账对外仍以**平台订单号**为主；内部通道信息仅管理端与运维可见（或脱敏给商户）。

### 4.2 收银台（Checkout）

- 用户看到的是 **支付产品**（微信、支付宝、…），**不是**「供应商 A / 供应商 B」。
- 用户点击「微信支付」后，由平台在服务端选定具体上游并完成调起/二维码/协议；**前端无需、也不应展示多条「微信」**。

### 4.3 商户台（Merchant Console）

- 面向商户运营与开发：**产品维度的说明、费率、能力、联调入口**。
- 若展示「通道级」信息，应明确为**平台侧路由结果**或**统计归因**（例如「本期微信走量」），避免与「多个微信产品」混淆；默认以**产品**为粒度展示即可。

### 4.4 管理后台（Admin）

- **配置支付产品**（名称、编码、对外展示顺序、是否启用）。
- **配置上游通道**（供应商、密钥、网关、健康状态、成本等）。
- **建立「支付产品 ↔ 上游通道」多对多/绑定关系**，并配置**路由策略**（权重、商户规则、熔断等）。
- **监控与报表**：既可按**产品**汇总，也可按**通道** drill-down，用于运维与对账。

---

## 5. 与数据模型、服务的对应关系（实现时对齐）

实现阶段需在库表与服务边界上显式区分，例如（名称可按项目实际命名）：

- **支付产品表**：产品编码、展示名、排序、状态。
- **上游通道表**：网关、密钥、商户号、费率成本、健康标记等。
- **产品与通道关联**：哪些通道属于哪个支付产品；是否启用；默认权重等。
- **订单 / 支付单**：记录**最终选用的 `channel_id`**，并保留**当时选用原因**（便于审计，可选）。

路由逻辑宜集中在 **Channel / Route 类服务**，下单与收银台只依赖「产品 → 路由 → 单条通道」的结果。

## 6. 实现状态（代码库）

### 6.1 代收与代付分轨

**代收（入金）** 与 **代付（出金）** 在数据与配置上**完全独立**：通道可只开代收、只开代付或两者都开；商户开通某通道的代收产品**不**自动获得代付能力。

| 方向 | 对外「产品」表 | 产品与通道绑定 | 商户白名单 |
|------|----------------|----------------|------------|
| 代收 | `pay_products` | `pay_product_channels`（仅权重与启用，**不含**费率） | `merchant_pay_products`（可选 `merchant_rate_bps` 覆盖商户默认代收费率） |
| 代付 | `payout_products` | `payout_product_channels`（同上） | `merchant_payout_products`（同上） |

订单侧仍以 `pay_products` / `pay_product_code` 等为主（代收链路）；代付出款若已建模，应对齐 `payout_products` 编码与路由（实现演进中见各服务）。

### 6.2 通道与费率层次

- **`channels`**：`supports_collect` / `supports_payout` 声明能力；`upstream_collect_rate_bps` / `upstream_payout_rate_bps` 为平台相对上游的费率（万分比）。
- **绑定行** `pay_product_channels` / `payout_product_channels`：仅维护路由（权重、启用），**不**存放费率。
- **`merchants`**：不再存放商户级默认费率；**商户—产品**授权行上的 `merchant_rate_bps`（及代付的 `fee_mode` / 固定费）为计费依据。

### 6.3 商户白名单（代收 / 代付各自一套）

对 **代收**：`merchant_pay_products` 规则不变——**若至少有一条启用的代收白名单**，则**严格模式**（仅白名单内产品可下单/收银台展示）；**若一条都没有**，则**开放模式**（与全平台「有可用代收通道的产品」一致）。

对 **代付**：`merchant_payout_products` 同理，独立计数；代收严格与否**不**影响代付开放/严格。

### 6.4 路由与接口（概要）

| 能力 | 说明 |
|------|------|
| 库表 | 见 `backend/deploy/sql/schema.sql`（推荐新环境直接建库；历史环境可参考 `migration_*.sql` 演进）。 |
| 代收路由 | `trade`：`ChannelsStore.Route` 等在选路时要求通道 `supports_collect`，并在 `pay_product_channels` 上加权。 |
| 代付路由 | 对称逻辑使用 `payout_*` 表与 `supports_payout`。 |
| 管理端 | 网关 Admin API：`/merchant-products` 与 `/merchant-payout-products`（前端同页组件 `payoutMode`）分别维护代收/代付产品与通道绑定。 |

**Demo 数据**：`seed_demo.sql` 含代收多通道权重示例及代付产品与商户授权示例；重建库时执行 `schema.sql` 后导入 `seed_demo.sql`。
