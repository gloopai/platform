🛠 模块一：服务端核心 (Backend RPC & Core Logic)
这是系统的“心脏”，优先完成数据建模和核心资金逻辑。

1.1 数据库与模型 (DB Schema)
任务：设计 MySQL 表结构。

关键字段：

merchants: 商户密钥、结算费率、白名单 IP、账户余额。

orders: 平台订单号、商户订单号、支付状态（待付/成功/失败/关闭）、通道 ID。

channels: 上游配置（网关 URL、公钥、私钥、权重、支持卡种）。

fund_logs: 变动流水（关联订单号、变动前余额、变动后余额、变动类型）。

1.2 核心 RPC 服务 (go-zero rpc)
Order Service：实现下单逻辑。必须包含分布式锁校验，防止商户重复提交。

Settle Service：处理“加钱”逻辑。要求在同一个事务内更新商户余额并记录 fund_logs。

Channel Service：路由分发。输入金额和类型，返回最合适的 ChannelID。

🔌 模块二：API 接入层 (Gateway & Interface)
这是对接商户和终端用户的“大门”。

2.1 商户 API (Merchant API)
下单接口：/v1/pay/order。接收商户参数，进行签名校验（验签失败直接拒绝）。

查单接口：/v1/pay/query。商户轮询支付状态。

2.2 回调网关 (Callback Gateway)
任务：接收上游支付公司的异步回调（Notify）。

逻辑：验证上游签名 -> 更新本地订单状态 -> 触发 NSQ 消息通知商户。

2.3 异步通知 (NSQ Consumer)
任务：独立的消费者进程。从队列取出支付成功消息，向商户服务器发送 Webhook。

策略：实现 5 次阶梯式重试逻辑。

🎨 模块三：收银台与官网 (Terminal & Web)
专注于用户交互体验和品牌展示。

3.1 收银台 (Checkout UI)
技术栈：Vue3 + Tailwind CSS。

功能：

解析支付指令：根据 API 返回的订单 ID，展示对应的支付名称、金额、倒计时。

动态拉起：检测微信/支付宝环境，点击“立即支付”唤起 App 或展示 Native 扫码框。

状态监听：长连接或轮询 API，支付成功后自动跳转到商户指定的 return_url。

3.2 官网门户 (Portal)
功能：静态展示页、产品优势、开发者文档入口。

📊 模块四：后台管理系统 (Management Admin)
提供给内部运营和商户自建使用的可视化工具。

4.1 总管理台 (Super Admin)
技术栈：Vue3 + Tailwind + Headless UI。

核心功能：

通道热切换：一键关闭故障上游。

费率配置：精细到每个商户、每个通道的扣费比例。

财务报表：导出每日总流水、净利润、商户提现审核。

4.2 商户后台 (Merchant Dashboard)
核心功能：

密钥管理：自主更换 API Secret。

对账中心：下载每日对账单，查看每一笔成功订单的扣费详情。

💡 给 AI 的执行指令模板 (按模块分发)
如果你现在开始第一个模块，可以这样对 AI 说：

“现在开始开发支付系统的**【模块一：服务端核心】**。

请使用 go-zero 框架，先生成 payment.rpc.trade 和 payment.rpc.core 的 .proto 定义。

数据库使用 MySQL，请提供支持高并发订单写入的 DDL 语句，订单表需要包含商户单号的唯一索引。

在 payment.rpc.trade 中实现一个下单逻辑，使用 Redis 保证商户订单号在 10 分钟内不被重复处理。”

等后端完成后，再执行第二个模块：

“现在开始**【模块二：API 接入层】**。

请定义 pay.api 文件，需要一个商户下单入口。

编写一个通用的签名校验 Middleware，要求商户按参数名排序后进行 MD5 签名。

实现一个 NSQ 消费者，负责在收到订单成功消息后，以阶梯重试的方式调用商户的回调 URL。”

产品经理的最后建议：
这种按模块完成的方式，联调测试非常重要。在完成【模块一】和【模块二】后，一定要让 AI 写一个**“模拟上游回调”**的脚本，测试整条链路是否能跑通加钱和发通知。
