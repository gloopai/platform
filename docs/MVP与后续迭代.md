# MVP 原则与后续深挖清单

本文给 **人与 AI 协作** 用：优先把**整条链路跑通**，单点功能**够用即可**，避免在某一模块过早做深。

---

## 1. 原则（务必遵守）

1. **先通后细**：以「管理台配置 → 商户下单 → 收银台支付 → 回调/查单 → 商户侧可见」等**主路径可演示**为第一目标。  
2. **最小可用**：每个功能点做到**能联调、能验收**即停；复杂交互、完整权限、精细报表、规则引擎等**一律后置**。  
3. **深挖单独记**：超出当前验收的设想，写入本文 **§3 后续深挖**，不在当期实现里「顺手」扩张范围。  
4. **文档同步**：完成一个 MVP 切片后，在 [`开发日志.md`](./开发日志.md) 写一行即可；**不必**为半成品写长篇设计。

---

## 2. 当前 MVP「跑通」大致包含（随仓库演进，以联调脚本与主流程为准）

**按步联调**：见 [**端到端联调一遍**](./端到端联调一遍.md)（建库 → `dev-up.sh` → 商户创建订单 → 收银台支付 → 模拟回调 → 交易查询）。

- 后端：订单创建、路由落通道、终端/收银台、回调与商户控制台基础读接口。  
- 管理端：通道 / 商户 / 支付产品与绑定 / 路由说明与概览统计等**已对接页**。  
- 商户端：登录、控制台汇总、开发联调下单、交易列表与详情（含支付产品展示名）。  
- 收银台：拉单、选产品、发起支付（演示）。

**不要求**在 MVP 阶段完成：全站订单管理、对账中心、风控、运维大屏、路由规则引擎、按成功率自动调度等。

---

## 3. 后续深挖（未做或仅占位，按需排期）

> 下列来自路线图与实现过程，**不承诺优先级**；开做前从业务价值与依赖（如可观测性）再选。

| 方向 | 内容 |
|------|------|
| 路由 / 通道 | **C4-rules** 规则引擎（金额段、时段、商户等级等）；**C4-metrics** 成功率/延迟驱动调度；**C4-ops** 与监控告警、人工切换审计；**A3** 熔断策略增强（Redis、自动恢复） |
| 网关 / 开放 API | **B2/B3** 查单/回调语义细化；**错误码体系**（产品不可用 vs 通道临时失败）；幂等与重放策略 |
| 管理端 | **C6** 全站用语与「产品/通道」模型彻底对齐；占位模块（全站订单、对账、结算、风控、运维等）**真实能力**与 RBAC |
| 商户端 | **D4** 控制台按**支付产品**维度的指标与图表；开放查单/退款的体验与文档深化 |
| 收银台 | **E3/E4** 环境检测与品牌化 |
| 财务与消息 | **F1～F3** 对账文件、差错闭环、NSQ 与订单状态全链路审计 |
| 工程化 | 压测、多机房、密钥轮换流程、前端组件库与无障碍全覆盖 |
