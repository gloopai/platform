# 基于本仓库的项目脚手架说明

本文档从 **pay-platform** 当前形态梳理出一套可复用的「全栈微服务 + **仅管理端（admin）**」脚手架约定，便于你新开项目时 **开箱即用**（复制结构、改名、删业务、接自己的领域逻辑）。

**本文范围**：只整理 **`frontend/apps/admin` 管理端** 与 **平台底座**（配置、RBAC、账户、监控等）。不包含其他前端应用；派生新项目时 **只保留 admin 即可**。

**分支 `scaffold/platform-admin`（本仓库）**：已 **删除** 非脚手架前端应用、支付业务文档与网关中的商户/OpenAPI/收银台及支付类 Admin API；`RegisterAdminHandlers` 仅保留 **登录、RBAC、后台用户、展示配置**；`seed_demo.sql` 仅含管理员与 RBAC 演示数据。与 `0.1` / `main` 的差异见根目录 `README.md`。

---

## 1. 仓库形态一览

| 区域 | 作用 |
|------|------|
| `backend/common/` | **共享 Go 模块**：Consul、gRPC/健康检查、JWT、配置、通知封装、部分领域无关工具等；各服务通过 `replace` 引用。 |
| `backend/services/gateway`、`backend/services/service-hub` | **网关 HTTP** + **平台 gRPC**（RBAC/管理员/展示配置等）；本分支无 core/trade 等支付服务。 |
| `backend/deploy/` | **SQL**（`schema.sql`、迁移、`seed`）、**初始化脚本**、运维/联调辅助脚本。 |
| `frontend/apps/admin/` | **唯一前端**：**Vite + Vue 3 + TypeScript + Tailwind + daisyUI**；根目录 `frontend/package.json` 用 **npm workspaces** 指向该包（见下节）。 |
| `frontend/shared/` | 管理端复用的 **UI 组件与约定**（如统一 Toast/Dialog）。 |
| 根目录 `dev-up.sh` | 本地一键拉起后端等；**脚手架建议**改为只跑 **`npm run dev`**（在 `frontend` 下仅针对 admin，见第 6 节）。 |
| `docs/` | 本分支保留 **`项目脚手架.md`**、**`管理端前端开发规范.md`**。 |

**基础设施（本分支）**：MySQL、Redis、Consul。**本分支不再启动** NSQ、core、trade、notice-consumer；网关只依赖 **service-hub** 与 Redis。

## 2. 技术栈摘要（便于选型对齐）

**后端**

- 语言：**Go**（本仓库常见 `go 1.25.x` 段，以各 `go.mod` 为准）。
- HTTP 网关：**go-zero**（`github.com/zeromicro/go-zero`），API 定义见 `gateway.api`，入口 `gateway.go`。
- RPC：**gRPC** + **Consul** 解析/注册（见 `backend/common/consulx`）。
- ORM：**GORM** + MySQL 驱动。
- 其他：JWT、Redis、NSQ 客户端等按服务引用。

**前端（仅 admin）**

- 构建：**Vite 5**。
- 框架：**Vue 3** + **Vue Router** + **TypeScript**。
- 样式：**Tailwind CSS 3** + **PostCSS** + **daisyUI**。
- **workspaces**：脚手架下根 `frontend/package.json` 建议写 **`"workspaces": ["apps/admin"]`**，scripts 只保留 admin 的 `dev` / `build`（例如 `"dev": "npm --workspace @platform/admin run dev"`；派生项目时按第 4 节改 scope）。

**工程约定（建议保留到新项目）**

- 网关 `internal/` 下 **handler / logic / middleware / svc** 分层。
- 前端业务页面按 **modules** 分目录，统一 API 封装与 `shared/ui` 交互组件（详见 [`管理端前端开发规范.md`](./管理端前端开发规范.md)）。

## 3. 推荐目录树（精简版）

新项目不必一开始复制全部 pay 业务，可按「最小骨架」理解：

```
repo-root/
├── README.md
├── dev-up.sh
├── .gitignore
├── backend/
│   ├── common/                 # 共享库（按需裁剪子包）
│   └── services/
│       ├── gateway/            # HTTP 入口
│       └── service-hub/        # 平台 gRPC（本分支）
├── frontend/
│   ├── package.json            # workspaces: ["apps/admin"]
│   ├── apps/
│   │   └── admin/              # 唯一前端 SPA
│   └── shared/                 # 可选：共享 UI / 工具
└── deploy/ 或 backend/deploy/  # SQL、初始化脚本
```

## 4. 从本仓库派生新项目时的「改名清单」

以下需 **全局替换** 或手动调整（路径按你新仓库名修改）：

| 类别 | 位置 | 说明 |
|------|------|------|
| Go module 路径 | 各 `backend/**/go.mod` | 例如 `github.com/gloopai/platform/gateway` → `github.com/<org>/<repo>/gateway` |
| `replace` | 各服务 `go.mod` | `replace .../common => ../../common` 相对路径一般不变，仅 module 名需一致 |
| `import` 路径 | 全部 `.go` 文件 | 与 `go.mod` 的 `module` 一致 |
| npm 包名 | `frontend/apps/admin/package.json` 的 `name` | 本仓库为 `@platform/admin`；派生时可改为 `@<scope>/admin` |
| 根 `frontend/package.json` | `workspaces` 与 `scripts` | 仅引用 `@.../admin` |
| 网关 `gateway.api` / 路由 | 按新领域删改 | 保留 `health`/`ready` 等运维接口更省事 |
| 配置 `etc/*.yaml` | 端口、Consul、DB、Redis、JWT 密钥等 | 与本地/环境一致 |
| `dev-up.sh` | 启动的服务列表、端口说明 | 与 `etc` 一致；前端段可只保留一条 admin 的 `npm run dev` |

**操作建议**：本仓库已定为 `github.com/gloopai/platform` 与 `@platform/admin`。拷贝到新仓库后，全局替换 module 与包名，再 `go mod tidy` 与各服务 `go build ./...`。

## 5. 裁剪建议：「最小可运行」组合

若希望脚手架 **更轻**（非完整支付域）：

1. **后端**：保留 `gateway` + **`service-hub`** + 精简版 `common`（本分支已按此裁剪）。
2. **前端**：**仅** `apps/admin`，根 `package.json` **只保留** admin 的 `dev` / `build`。
3. **infra**：MySQL + Redis + Consul；本分支网关 **不** 强依赖 NSQ。
4. **数据库**：用 `schema.sql` 的思路保留「迁移 + seed」流程，表结构换成你的领域。

---

## 6. 本地开发流程（沿用本仓库习惯）

1. **准备依赖**：MySQL、Consul、Redis — 与 `dev-up.sh` 一致。
2. **初始化库表**：参考 `backend/deploy/init_demo.sh` 及其中 SQL（新项目替换为你的 schema）。
3. **启动后端**：`bash dev-up.sh`（或只启动你保留的服务；日志在 `.dev-logs/`）。
4. **启动管理端**：在 `frontend/` 下 `npm install`，执行 **`npm run dev`**（需与根 `package.json` 中指向 admin 的 script 一致）；Vite `proxy` 指向网关 **Admin** 端口（见 `apps/admin/vite.config.ts`）。

## 7. 可选：物理脚手架仓库的两种落地方式

| 方式 | 做法 |
|------|------|
| **A. 模板仓库** | 复制本仓库后执行第 4 节改名，再删掉 `internal` 里支付相关业务，保留空壳路由与健康检查。 |
| **B. 子目录模板** | 在组织内单独建 `pay-platform-scaffold` 仓库，只提交上述「最小骨架」+ 占位 README，业务从空接口开始写。 |

两种方式都建议把 **`.gitignore`** 与 **`dev-up.sh` 模式** 固定下来，避免每个新项目重复踩坑。

## 8. 基础平台模块（系统配置 / 角色 / 菜单 / 账户 / 监控）

以下模块属于 **多数后台系统都会用到的「底座」**，本仓库已有一套可随脚手架一起带走或裁剪的实现。派生新项目时，建议 **整块保留** 再在其上长业务表与路由；若做极简 demo，至少保留 **登录 + health + 一张用户表** 再扩展。

### 8.1 能力对照

| 模块 | 职责摘要 | 本仓库落点（便于复制） |
|------|----------|------------------------|
| **系统配置** | 全局键值（如展示类配置）；管理端读写 `display_settings` | 表 `global_settings`；接口 `GET/PUT /v1/admin/display_settings` |
| **角色** | 角色 CRUD、角色与菜单/权限点绑定 | 表 `admin_roles`、`admin_role_menus`、`admin_role_permissions`；路由前缀 `/v1/admin/rbac/roles`、`.../roles/:id/menus`、`.../roles/:id/perm_keys` |
| **菜单** | 树形菜单、分组、侧栏/头像区 placement；登录后「我的菜单」 | 表 `admin_menus`；`GET /v1/admin/rbac/my_menu`、菜单 CRUD 等 |
| **权限与接口规则** | 功能点（`perm_key`）、HTTP 方法 + path 模式映射到权限（免改代码调权限） | 表 `admin_permissions`、`admin_api_rules`；对应 `/v1/admin/rbac/permissions`、`/v1/admin/rbac/api_rules`；网关侧有 **Admin RBAC 中间件** 做鉴权（见 `internal/middleware/adminrbacmiddleware.go`） |
| **账户（后台用户）** | 登录登出、当前用户、用户 CRUD、重置密码、MFA | 表 `admin_users`、`admin_user_roles`；`POST /v1/admin/login`、`/logout`、`GET /v1/me`；`/v1/admin/admin_users` 系列；实现见 `internal/handler/admin/auth.go` 等 |
| **监控 / 运维** | 探活、就绪、依赖检查、服务实例列表（Consul） | **无鉴权**：各服务可暴露 `GET /health`（见 `internal/handler/health.go`）。**Admin 端口**：`GET /ready` 聚合 gRPC/Redis 等检查（`internal/handler/ready.go` + `backend/common/healthx`）；**需登录**：`GET /v1/admin/ops/services` 拉 Consul 中注册服务状态（`internal/logic/admin/ops.go`，配置 `etc/gateway-api.yaml` 里 `OpsMonitor`）。管理端页面：`frontend/apps/admin/src/views/modules/ops/OpsPage.vue` |

**RBAC 管理界面**（菜单 / 功能点 / 接口规则 / 角色授权 / 后台用户）集中在：`frontend/apps/admin/src/views/modules/rbac/`。

**数据表定义**：`backend/deploy/sql/schema.sql` 中 `admin_users`、`global_settings`、`admin_roles`、`admin_menus`、`admin_user_roles`、`admin_role_menus`、`admin_permissions`、`admin_role_permissions`、`admin_api_rules`；演示种子数据可参考 `backend/deploy/sql/seed_demo.sql` 里 RBAC 相关 `INSERT`。

**路由注册**：`backend/services/gateway/internal/handler/routes.go` 的 `RegisterAdminHandlers`（平台接口如上）；`/ready` 与 `/v1/admin/ops/services` 在 `gateway.go` 中额外挂载到 Admin Server。

### 8.2 派生新项目时的建议

1. **优先保留**：`global_settings` + 整套 `admin_*` RBAC 表 + 对应 gateway handler/logic（可按业务删掉不需要的 `perm_key` 与 `admin_api_rules` 行）。
2. **监控**：保留 `healthx` 与 `ReadyHandler` 模式，便于 K8s/负载均衡探针；`ops/services` 依赖 **Consul**，若新项目不用 Consul，可改为静态列表或调用各服务 `/health`（需自行改 `ops` 逻辑）。
3. **前端**：`modules/rbac` 与 `modules/ops` 与后端契约紧耦合，改名时注意 **API 路径与权限键** 一并调整。

## 10. 小结

本项目的「脚手架价值」主要在于：**Go 多模块 + common 复用 + go-zero 网关分层 + 仅 admin 前端 + 一键 dev 脚本 + 部署 SQL 流程**，以及第 **8** 节中的 **系统配置、RBAC、后台账户、探活与运维监控** 等可复用底座。新开项目时按第 4 节改名、按第 5 节裁剪，即可在 **同一套工程习惯** 上快速落地业务，而不必从零拼凑目录与依赖。

若你希望下一步做成 **独立 git 仓库** 或 **Cookiecutter / 交互式 shell 脚本** 生成空项目，可以在现有结构上再加一层生成器（本文不展开，需要时可单独拆任务）。
