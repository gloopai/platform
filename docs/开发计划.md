第一阶段：基础设施与核心模型 (Week 1: Foundations)
目标：建立微服务骨架，确保存款流程打通。

1.1 环境搭建：

配置 go-zero 工程目录，集成 Consul 服务注册中心。

搭建 NSQ 消息队列环境，用于异步通知。

1.2 数据库建模 (DB Schema)：

商户表：merchants (API密钥、状态、费率)。

通道表：channels (上游网关、商户号、密钥、权重)。

订单表：orders (唯一索引、金额、状态机、支付成功耗时)。

资金流水表：fund_logs (变动前/后余额，用于财务审计)。

1.3 核心 RPC 开发：

Order.Create：支持幂等性校验，防止重复下单。

Channel.Route：初步实现根据配置权重选择上游通道。

第二阶段：支付链路与协议适配 (Week 2: Payment Gateway)
目标：完成“商户->平台->上游”的闭环，实现异步回调。

2.1 上游适配器 (Adapter Pattern)：

开发统一的 PayHandler 接口，对接首批 2-3 个上游支付通道。

实现签名/验签通用工具类（支持 RSA, MD5, HS256）。

2.2 收银台 H5 开发 (Tailwind CSS)：

实现响应式页面：展示支付金额、选择支付方式（微信/支付宝/网银）。

支付状态轮询机制：前端自动感应支付成功并跳转。

2.3 异步通知引擎 (NSQ)：

编写 Notice-Consumer：接收支付成功消息，按 15s/1m/5m/30m 阶梯频率推送到商户 Webhook。

第三阶段：管理中枢与财务对账 (Week 3: Admin & Finance)
目标：实现可视化管理，解决“差错账”和“分润”问题。

3.1 总管理台 (Vue3 + Element Plus/AntD)：

看板：今日成交额、订单成功率曲线、上游通道实时状态。

配置：动态修改商户费率、切换上游网关。

3.2 对账系统 (Reconciliation)：

定时任务：每晨自动拉取上游对账单。

三方勾对逻辑：比对（上游流水 vs 平台订单 vs 商户余额），发现差错订单自动挂起异常告警。

3.3 结算逻辑：

支持 T1 (隔日) 与 D0 (实时) 结算逻辑。

自动扣除平台服务费，更新商户可提现余额。

第四阶段：安全加固与生产上线 (Week 4: Security & Scaling)
目标：应对高并发压力，确保系统“抗造”。

4.1 深度风控 (Risk Control)：

实现单 IP 频率限制、单笔金额上下限。

敏感操作（如修改提现地址、修改费率）强制双人审核记录。

4.2 性能优化 (Stability)：

Redis 分布式锁：在结算加钱环节强制加锁，杜绝并发坏账。

日志体系：集成 Loki 或 ELK，全链路追踪每个 TraceID 的支付耗时。

4.3 文档站与沙箱：

自动化生成 Swagger 文档。

提供 Test Mode：商户使用测试密钥即可模拟支付回调。
